本 Art Bible は、brainstorm Phase 4 で art-director が提示し、ユーザーが B+A Hybrid として承認した **「ポップ・ダンジョン + オーブ・ルミナ」** Hybrid を起点として拡張する。

## 1. Visual Identity Statement

### One-Line Visual Rule

> ***"Every sprite lives in a world of stickers and sunshine — and the moment a class switches, that world exhales the orb's color and remembers who you are."***
>
> （日本語意訳: すべてのスプライトはステッカーと陽光の世界に生きる — 職業が切替わる瞬間、世界はオーブの色で息を吐き、あなたが誰かを思い出す）

**実用版（制作指示用の短縮）**:
> *"Sticker-sheet sprites + sunlit world. On switch: the world briefly breathes the new orb's color."*

この 2 文は **すべてのアート判断** に対して「合致するか／反するか」の裁定を下す。

### 4 Supporting Principles（各 Principle はピラーに 1:1 対応）

#### Principle 1: **Dual-Read Silhouette（二重読みシルエット）** → Pillar 3「可愛いけど、歯ごたえある」

キャラのシルエットは「**職業の種別**」と「**いまの行動意図（戦闘 / 待機 / 移動）**」を同時に 3 秒で伝える。剣士の攻撃シルエットは重心低く鋭角、待機シルエットは背中を丸め可愛い。

- **Design Test**: 10 枚のアニメーションフレームをシルエットだけ並べたとき、「重いフレーム（戦闘）」と「かわいいフレーム（待機/勝利）」がパッと分類できるか？
- **Pillar 3 との接続**: 「戦闘モーションは重く、表情・待機ポーズ・勝利演出はかわいく」のピラーテストを視覚レイヤーで体現する原則

#### Principle 2: **Orb-Gate Visual Language（オーブ・ゲート視覚言語）** → Pillar 2「一歩ごとに、次の鍵が見える」

まだ取れない職業で開く **扉・障壁・ギミック** は、その職業のアクセントカラーを **彩度 50% 以下** で環境に薄く埋め込む。新オーブ取得時は 3 秒間「画面内に散る光の粒子」演出で探索欲求を刺激する。

- **Design Test**: プレイヤーが扉の前で「この色は弓士オーブだ（Amber）」と初見で気づけるか？（5 人テスト中 4 人以上）
- **Pillar 2 との接続**: 「次の鍵で何が開くかが常に頭にある」状態を、環境に埋め込む色ヒントとして視覚的に担保

#### Principle 3: **Class-Switch Color Wash（切替カラーウォッシュ）** → Pillar 1「切替が、花になる」

切替瞬間、画面四隅から内側へ **0.1〜0.2 秒の放射状グラデーション** が新オーブ色で広がってフェードアウト。さらに切替エフェクトの「花びら型パーティクル」は **職業ごとに形が異なる**：
- 剣士 = 刀身の欠片（鋭角、直線的）
- 弓士 = 矢羽（細長、風を切る）
- 魔法使い = 魔法陣片（幾何学的、曲線）

- **Design Test**: 音を消してプレイしても、「今どの職業に切替えたか」がエフェクトの **形と色** で判別できるか？
- **Pillar 1 との接続**: 「切替そのものが視覚・聴覚の報酬」。報酬の視覚言語を **色（ウォッシュ）× 形（パーティクル）** の二層で定義

#### Principle 4: **Combo-Brightness Escalation（コンボ輝度エスカレーション）** → Pillar 4「1職業で深く、全部でもっと深く」

コンボ継続中、エフェクトの輝度と粒子密度が **段階的に上昇** する。
- 単独職業連続攻撃 = 基準輝度・単色の光
- 2 職業切替コンボ = 2 色の光が混ざり始める
- 3 職業切替コンボ = 3 色が衝突し、白光に昇華する「祭り」状態

数値目安（プレイテストで調整前提）: 2 職業コンボ **+20% 輝度**、3 職業コンボ **+40% 輝度**。ただし読み取り不能な輝度崩壊は避ける。

- **Design Test**: 1 職業連打と 3 職業コンボを録画で比較したとき、後者が「明らかに派手」と感じるか？かつ、画面の情報可読性（敵位置・自機位置）は保たれているか？
- **Pillar 4 との接続**: 「単独でも成立、組合せで爆発」を視覚のリッチさの段階的上昇として体現

---

### 保留事項（将来レビュー時に再検討）

1. **One-Line Rule の詩的表現**: ポエム版と実用版を両方併記した。現場で実用版のみ使って不足あれば上位ルール（詩的版）に立ち戻る運用を試す
2. **Principle 4 の数値（+20%/+40%）**: プレイテストで要調整。数値固定より「段階的に派手」方針を優先
3. **アンチピラーの視覚制約**（暗さ上限、暴力表現禁止、セクシャル表現禁止）: Section 9 Style Prohibitions に切り出す予定

## 2. Mood & Atmosphere

各 Game State ごとに、ムードと照明の目標を定義する。参考比較は「作り手・実装者がイメージを共有するため」の例示。

### State 1: Exploration / Traversal（歩いている時）

- **例え**: メープルストーリーのフィールド、明るい昼間のダンジョン
- **Primary Emotion**: 穏やかな好奇心と発見予感 — 「次の曲がり角に何かある」
- **Lighting**: 午前〜正午の柔らかな斜光、色温度 Warm (5500〜6500K)、コントラスト Low-Med。左上からの平行光源、背景遠景は明度を落として奥行き演出（Depth Haze）
- **Atmospheric Descriptors**: sunlit / airy / curious / inviting / wandering
- **Energy Level**: measured
- **Key Visual Element**: Light Shaft（光の筋）+ 薄く配置された未取得オーブのアクセントカラー（目の端に引っかかる、Principle 2 直結）

### State 2: Combat（通常戦闘）

- **例え**: 白猫プロジェクトの戦闘、マリオで敵を踏む爽快感
- **Primary Emotion**: 集中と快感の同居 — 「やれている」高揚感
- **Lighting**: 昼の鮮明な直射光、色温度 Neutral-Warm (5000K)、コントラスト Med-High。キャラとエフェクトの発光色が環境光を食い始める動的状態
- **Atmospheric Descriptors**: sharp / kinetic / bright / crisp / rhythmic
- **Energy Level**: elevated（コンボ継続でさらに上昇）
- **Key Visual Element**: Impact Spark（ヒット火花）+ コンボ輝度の段階上昇（Principle 4 直結）

### State 3: Class Switch Moment（職業切替、0.1〜0.4 秒）★本作の目玉

- **例え**: ドラゴンボールの変身バンク、原神のキャラ切替エフェクト
- **Primary Emotion**: 全感覚への短い衝撃 — 「世界が変わった」という確信
- **Lighting**: 瞬間的な全画面発光、新オーブ固有色に支配される。剣士 = Electric Blue、弓士 = Amber、魔法使い = Arcane Violet（Section 4 で正式定義）。アンビエント光が一瞬全て新オーブ色に染まる
- **Atmospheric Descriptors**: explosive / crystalline / momentary / charged / decisive
- **Energy Level**: frenetic（持続時間 0.1〜0.4 秒）
- **Key Visual Element**: 画面四隅からの放射状 Color Wash（Principle 3）+ 職業固有形状パーティクル（剣士 = 刀身の欠片、弓士 = 矢羽、魔法使い = 魔法陣片）。この瞬間のみ他 visual element が一時的にフラッシュアウトして切替が唯一の主役

### State 4: Boss Battle

- **例え**: Hollow Knight のボス戦、劇場のスポットライト舞台
- **Primary Emotion**: 緊張とスペクタクルの共存 — 「大きなものと向き合っている」
- **Lighting**: 劇場的照明、ボスが発光源を持つ（Rim Light）、アリーナ環境照明は Exploration より 20〜30% 暗く（ただし「暗い洞窟」ではなく「スポットライトの当たるステージ」）。コントラスト High
- **Atmospheric Descriptors**: theatrical / tense / grandiose / luminous / confrontational
- **Energy Level**: elevated → frenetic（フェーズ変化時に跳ね上がる）
- **Key Visual Element**: ボスの Glow Aura + 背景呼応演出（床に文様浮かぶ、壁が脈打つ）。フェーズ変化時は環境光の色温度を瞬時に +1000K シフト

### State 5: Victory / Kill Confirm（敵撃破の瞬間）

- **例え**: スーパーマリオで敵を踏んだ時の「プッ」、でもより派手
- **Primary Emotion**: 爽快なカタルシス — 「決まった」
- **Lighting**: 瞬間的な明度スパイク（0.1 秒）→ 素早く Exploration 照明復帰。倒した敵のドロップ光のアクセントカラーで一瞬染まる
- **Atmospheric Descriptors**: punchy / satisfying / bright / clean / resolving
- **Energy Level**: elevated → measured（撃破後の余韻）
- **Key Visual Element**: 敵の解散エフェクト（スプライトがステッカーを剥がすように分解、光の粒が散る）。コンボ中 Kill は Principle 4 の輝度最高点で炸裂

### State 6: Defeat / Game Over（死亡）

- **例え**: 軽いリトライ（重いゲームオーバー演出は意図的に避ける）
- **Primary Emotion**: 悔恨と「もう一度」の衝動 — ポジティブな悔しさ
- **Lighting**: 色が抜けて Desaturated（彩度 30〜40% に低下）→ **画面が白っぽくフェード**。色温度 Cool (4000K)、コントラスト Low。**「暗い」ではなく「色が薄まった」表現**（Anti-Pillar 遵守）
- **Atmospheric Descriptors**: washed-out / quiet / still / deflating / brief
- **Energy Level**: still
- **Key Visual Element**: Unity-chan の紙吹雪デスアニメーション（戦闘爽快感と対称の「軽い解散」感）→ ホワイトアウト → Retry UI
- **Distinguish**: Bench（暖色の静けさ）と真逆の脱彩色。ただし暗さ・重さ・ホラーは排除

### State 7: Orb Acquisition（新職業オーブ獲得の瞬間）★本作の最高潮

- **例え**: ゼルダのハートのかけら取得 × 10 倍派手
- **Primary Emotion**: Peak 発見の歓喜 — 「世界が広がった」
- **Lighting**: 全画面発光の最高輝度点、新オーブ固有色で画面が 1〜2 秒間 wash される（Switch Moment の Color Wash を「より長く・より豊か」に展開）。コントラスト High → 3 秒で Exploration 照明に収束
- **Atmospheric Descriptors**: euphoric / radiant / expansive / ceremonial / unforgettable
- **Energy Level**: frenetic（最初の 0.5 秒）→ contemplative（余韻の 2〜3 秒）
- **Key Visual Element**: オーブが Unity-chan の胸元に溶け込む演出 + 新オーブ色の光の粒子が 3 秒間漂う（「この色の場所がこれから行ける」直感的報酬）
- **Distinguish from State 3**: 切替 = 繰り返せる呼吸（点）、獲得 = 一度きりの祝祭（面）。持続時間と格で差別化

### State 8: Bench / Rest / Save Point

- **例え**: ポケモンセンター、ルーンファクトリーの家に帰った安心感
- **Primary Emotion**: 安らぎと回復 — 「ここは安全、ひと息つける」
- **Lighting**: 夕暮れの暖かな光（4000〜4500K Warm Orange）、コントラスト Low。光源は小さな魔法の炎 or 光る植物（柔らかい内部光源）、周囲より明度が落ちてセーブポイントが画面の最も暖かい点に
- **Atmospheric Descriptors**: cozy / amber / sheltered / breathing / gentle
- **Energy Level**: contemplative
- **Key Visual Element**: ベンチに座った Unity-chan のアイドルアニメ（深呼吸、髪を直す、オーブを手でいじる）+ 蛍のような小さな光の粒 + セーブ完了時の温かい光の波紋 micro-reward

### State 9: Menu / Title

- **例え**: 普通にアニメのオープニング画面っぽい
- **Primary Emotion**: 期待感と親しみやすさ — 「また会えた」
- **Lighting**: 昼の明るさベース、柔らかな Warm Light (6000K)、コントラスト Med
- **Atmospheric Descriptors**: welcoming / iconic / clean / hopeful / composed
- **Energy Level**: measured（Loop アニメーション）
- **Key Visual Element**: タイトルロゴ中心 + Unity-chan が主な職業オーブを手元で弄ぶ短ループ + 背景の職業色が周期的にゆっくり切り替わる（「このゲームは色で語る」予告）

### State 10: Locked Door / Gated Area（未開放ゲート）★Pillar 2 の核

- **例え**: ドラクエの「鍵が必要」な扉、Hollow Knight の能力ゲート
- **Primary Emotion**: 悔しい好奇心 — 「いつかここを開ける」という宿題感
- **Lighting**: 扉・障壁周辺のみ局所的に彩度低下（Desaturation）。周囲の環境は通常の Exploration 照明のまま、ゲート部分だけが「くすんで見える」。オーブカラーは彩度 **40〜50% 以下** にしてうっすらゲート表面に滲む
- **Atmospheric Descriptors**: muted / teasing / foreshadowing / patient / purposeful
- **Energy Level**: measured
- **Key Visual Element**: くすんだオーブカラーの Glow Outline（彩度 40〜50%）がゲートの縁に走る → プレイヤーが「この色は弓士オーブだ」と初見で気づける → 後の Orb Acquisition 時に「鮮やかに覚醒」することで before/after の発見感を増幅

### コントラストマップ（設計健全性）

| | ← 完了・安全・静 → | | ← 発見・変化・絶頂 → |
|---|---|---|---|
| **frenetic** | | | State 3 (Switch), State 7 (Orb Acq.) |
| **elevated** | State 5 (Victory) | | State 4 (Boss) |
| **measured** | State 9 (Menu), State 10 (Locked) | | State 1 (Exploration) |
| **contemplative** | State 8 (Bench) | | State 2 (Combat*, 動的上昇) |
| **still** | State 6 (Defeat) | | |

*State 2 Combat は Measured 〜 Elevated の間で動的に変化

**設計健全性チェック**:
- Energy 幅: still 〜 frenetic まで全段階カバー
- frenetic 2 件 (State 3, 7) は **持続時間で差別化**（点 vs 面）
- elevated 2 件 (State 4, 5) は **感情トーン方向で差別化**（Tense vs Resolving）
- still 1 件 (State 6 Defeat) は **意図的孤立**（唯一の脱彩色体験）

---

### 保留事項

1. **State 6 のホワイトアウト演出**: Anti-Pillar（ホラー/暗さ回避）遵守で採用。将来プレイテストで「軽すぎる」フィードバックがあれば再検討
2. **State 4 の色温度 +1000K シフト**: 仮数値、Section 4（Color System）でボス固有色確定時に再調整
3. **State 10 の彩度 40-50%**: Principle 2 の記述と一致。プレイテストで「気付きにくい」なら彩度上限を 55-60% まで緩める余地

## 3. Shape Language

### A. Character Silhouette Philosophy

**基本方針**: すべての職業キャラは「ステッカーシートから切り取られた形」として設計。縁取りで完結した輪郭、職業ごとに異なる重心と武器アウトライン、48×48px サムネイルでも「頭部 + 武器のシルエット二点セット」で職業が判別できる。

**参照**: MapleStory の職業シルエット（騎士の肩鎧、魔法使いの杖で瞬時識別）× Mega Man X の武器シルエット（Arm Cannon でロックマンとわかる）。

#### 職業別 Identifying Trait

| 職業 | 頭部 | 武器 | スタンス重心 | 衣装アウトライン | ゲーム比較 |
|------|------|------|-------------|----------------|-----------|
| **剣士** | ヘルメット or 短い前髪（横張り） | 大剣が肩より上に縦長ライン | 低重心・前傾 | 台形「広肩」 | Mega Man X のX アーマー |
| **弓士** | ロングポニーテール or 耳当て（後方流れ） | 弓の弧が体外に半円 | 高重心・後重心 | S 字曲線の細身 | Ori の Ori |
| **魔法使い** | とんがり帽子（頭頂上伸） | 杖 or 魔法陣が頭上浮遊 | 中立・直立、足幅狭め | 逆三角形に広がるローブ | MapleStory Arch Mage |
| **Unity-chan 共通ベース** | オレンジツインテール（左右対称） | 武器差替で上書き | 直立やや前傾 | 公式素材のアニメ調 | — |

**Design Test**: 各職業のアイドルポーズを 48×48px モノクロシルエットに変換したとき、**3 秒以内に職業が判別可能**。判別基準は「頭部形状または武器シルエットのいずれか一方が機能すれば合格」。

*仕えるピラー: Pillar 3（Cute & Crunchy — 戦闘フレームは重く低重心、待機フレームは可愛く丸み）、Pillar 2（次の鍵 — 職業識別が瞬時にできることで「このゲートには剣士が要りそう」という読みが生まれる）*

### B. Environment Geometry

**基本方針**: 環境は **「直線と角」を骨格**にしつつ、**生命の存在を示す有機的アクセント**（苔、ツタ、粒子の光）を重ねる「**廃墟の幾何学 + 自然の侵食**」構造。ゾーンごとに支配的な形を変えるが、全ゾーン共通で **「プラットフォームは水平直線」「壁は垂直直線」** を守る。

**参照**: Hollow Knight の廃墟感（石造り幾何学）× MapleStory のエリア別有機的フレーバー。Ori のような「有機曲線が支配する環境」とは対照的に **「幾何学が主、有機が従」**。

#### ゾーン別バリエーション

| ゾーン | 支配形 | アクセント形 | 機能的意味 |
|-------|------|------------|----------|
| **ゾーン1（王国遺跡）** | 四角・長方形の石積み、水平プラットフォーム | 崩れた石のギザギザ、ツタの曲線 | 入門：形が整って操作習得しやすい |
| **ゾーン2（自然洞窟）** | 不等辺三角形の鍾乳石・岩床、段差不均一 | 発光キノコの円形、水面の水平線 | 中盤：形の不規則さが難度上昇を示す |
| **ゾーン3（古代神殿）** | 幾何学文様の床・円弧状の柱 | 魔法陣フラグメントの多角形、宙の光三角 | 終盤：人工物極まる、Pillar 4 の深度を形で暗示 |

#### 部屋間接続（メトロイドヴァニア遷移）

- **横スクロール接続** = 画面端の **アーチ型開口部**（全ゾーン統一）
- **縦スクロール接続（上下）** = 天井/床の **丸みを帯びた穴**（四角い穴 = 落下穴 との混同回避）
- **ゲート（職業ロック）** = **六角形の封印紋章**、Pillar 2 のオーブカラーで縁取る

**Design Test**: ゾーン1 と ゾーン3 の部屋スクリーンショットを並べたとき、「形の複雑さが増している」と感じられる。かつどちらでも **「プラットフォームの水平線」が 3 秒以内に読める**。

*仕えるピラー: Pillar 2（ゲートの六角形 + オーブカラーで「ここには職業が要る」を形で宣言）、Pillar 3（有機アクセントの可愛さ + 形の難化で cute and crunchy 両立）*

### C. UI Shape Grammar

**基本方針**: UI は world aesthetic と **半分溶け込み、半分突き出る**。世界と同じ形語彙（丸みを帯びたステッカー風）を使いつつ、輪郭線と塗りの彩度を環境より高くして HUD が自己主張しすぎず埋没もしない。MapleStory の「世界に馴染む可愛い UI」を目指す。

#### 主要 UI 要素の形状

| UI 要素 | 形 | 根拠 |
|--------|-----|-----|
| **オーブ HUD（職業切替インジケーター）** | **六角形オーブアイコン** × 現在職業は大、待機職業は小。縁取りが職業色で光る | ゲート紋章と同じ六角形で「**鍵＝職業**」の連想強化（Pillar 2） |
| **HP バー** | 左端揃え横長の丸角矩形。端が完全半円ではなく「切れた丸角」で手作り感 | 完全丸角は UI っぽすぎる、ステッカー的不完全さで溶け込む |
| **職業切替インジケーター** | 現在職業オーブが小さく跳ねる + 次職業オーブが輝く | 静的矢印より「オーブが生きている」感、Pillar 1「切替が花」を予告 |
| **ダメージ数値ポップ** | 丸みある太ゴシック体、1〜2px 黒縁取り | MapleStory の「ポコン」と出る数値の哲学 |
| **ゲートアイコン（扉表示）** | 六角形 + 職業シルエットのシャドー | オーブ HUD と形統一、「この形 = 職業概念」をプレイヤー学習 |
| **マップ UI（将来）** | 四角タイルの集合体、部屋の形を正方形グリッドで | Hollow Knight のマップ踏襲、環境幾何学と同じ角形言語 |

**UGUI → UI Toolkit 移行時**: Tier 2 で UI Toolkit に移行しても「六角形オーブ」「丸角バー」の視覚仕様は固定。技術実装は変わっても形語彙は不変。

**Design Test**: HUD 表示したまま戦闘中画面を 3 秒見せたとき、「**プレイヤーキャラ位置 → HP → 現在職業**」の 3 つが順番に読める。

*仕えるピラー: Pillar 2（六角形 = 職業の形語彙統一で UI とゲート世界が連動）、Pillar 1（切替インジケーターが跳ねることで切替欲求を形で煽る）*

### D. Hero Shapes vs Supporting Shapes

**視線誘導の原則**: **「尖った形は目を引く、丸い形は目を受け入れる」**（Gestalt 図地分離）。プレイヤーが「次に見るべきもの」は **鋭角または輝く形**、「後退すべき背景」は **丸みと低彩度の形**。

#### Hero Shape（前景・注目・アクション）— 鋭角、縁取りあり、高彩度、動く

| 要素 | 形の特徴 | 機能 |
|------|--------|-----|
| Unity-chan（プレイヤー） | 武器シルエット鋭角突出、黒縁取り 2-3px | 最前景の優位性 |
| 敵キャラ | 不規則ギザギザ or 明らかに尖った角 | 「危険」を形で宣言 |
| 職業ゲート・扉 | 六角形の鋭い角 + オーブカラー Glow | 地図的ランドマーク |
| ヒットエフェクト・切替パーティクル | 星型・矢印型・鋭い破片 | 「何かが起きた」を形の衝撃で伝える |
| オーブアイテム（取得前） | 多角形（6-8 角）+ 内側発光 | 「これを取れ」の磁力 |

#### Supporting Shape（後景・環境）— 曲線 or 緩い角、薄縁 or 縁なし、低彩度、静止

| 要素 | 形の特徴 | 機能 |
|------|--------|-----|
| 背景遠景タイル | 丸角の石、ゆるいアーチ | 奥行き演出、視線奪わず |
| 地面・床プラットフォーム | 水平直線 + 上面微テクスチャ | 「踏める」を宣言しつつ後退 |
| 環境小物（ツタ・キノコ） | 有機曲線、楕円 | 温かみを添え、邪魔しない |
| 未発見エリア（マップ） | 完全曲線、角なし | 「まだ読めない場所」の曖昧さ |

#### レイヤー順序（視線誘導の実装）

1. プレイヤーキャラ（最前景、鋭角優位）
2. 敵・インタラクタブル（競合）
3. ゲート・宝箱（Hero、静止）
4. プラットフォーム（水平直線、後退）
5. 背景建築物（丸角低彩度、奥行き）
6. 最遠景（曲線のみ、完全後退）

**Design Test**: 戦闘シーンの静止画からプレイヤーキャラを除いたとき、視線が「**敵キャラ → ゲート/宝箱 → 背景**」の順に向かう。5 人テストで確認。

*仕えるピラー: Pillar 2（ゲートを Hero Shape にすることで「あそこが気になる」が自然発生）、Pillar 3（敵の鋭角 = 手ごわそう、Unityちゃんの可愛い丸み = cute、両立で cute vs crunchy 実現）*

### 形語彙の統一辞書（Section 5-7 の参照用）

| 形 | 意味 | 使う場所 |
|----|-----|--------|
| **六角形** | 職業・オーブの概念 | ゲート、UI オーブアイコン |
| **鋭角三角 / 破片** | 衝撃・切替・攻撃 | エフェクト、敵シルエット |

| **アーチ形開口** | 部屋接続・通過点 | ドア、ゾーン遷移 |

### D. Per-Zone Color Temperature Rules

#### Zone 1: 王国遺跡（入門）
- **色温度**: Warm 5500〜6500K（晴れた日中）
- **支配色**: Parchment + Moss Stone アクセント
- **照明色**: Sky Ivory 寄りの白〜薄黄（「窓から差す昼の光」）

- **禁止色**: Deep Slate 大面積（洞窟 Zone 2 との差別化）、Void Indigo 環境使用（Zone 3 予約）、低彩度モノクロ背景（Sunlit Pillar 違反）

- **色温度**: Cool-Neutral 4000〜5000K（地下拡散光）
- **支配色**: Deep Slate + 発光キノコ Emerald 点描
- **照明色**: 冷たい青白 + 生物発光の緑
- **ムード比較**: Hollow Knight の菌類の廊下、ただし明度管理で真っ暗にしない
- **禁止色**: Sky Ivory 大面積、Ember Crimson 環境 Glow（剣士ゲート専用）、**明度 15% 以下の真っ暗（Anti-Pillar 遵守）**

#### Zone 3: 古代神殿（終盤）
- **色温度**: Cool + Atmospheric 3500〜4500K + 高色域
- **支配色**: Deep Slate + Void Indigo 魔法陣 Glow
- **照明色**: 深インディゴ〜紫の環境光 + 魔法陣床の発光
- **ムード比較**: 原神の古代神殿エリアを 2D 横スクに圧縮
- **禁止色**: Parchment 大面積（Zone 1 と混同）、Moss Stone 植生（Zone 2 引継）、Warm Shadow 照明（ベンチ専用）

**ゾーン遷移の原則**: アーチ形開口部の内側で色温度が 1〜2 秒かけてクロスフェード。急な明暗遷移は禁止（Sunlit 連続性保護）。

#### Backup Cue（必須対応）

- **Cue C パーティクル**: Section 1 Principle 3 の形言語で既定（剣士＝刀身の欠片、弓士＝矢羽）

**危険ペア 2: 剣士(赤) vs 魔法使い(紫) @ Protanopia**

- **Cue B モーション**: ゲート Glow 点滅速度差別化（剣士 2Hz 鋭い点滅、魔法使い 0.5Hz 緩い脈動）

**危険ペア 3: 剣士色 vs HP Danger 色**
- **解決**: 明度差 10%（HP Danger 65% vs 剣士 55%）+ HP Low 点滅アニメーションで動きで区別

## 5. Character Design Direction

### A. Visual Archetype

#### Unity-chan（プレイヤーキャラクター）

**ベースライン**: Unity-chan 公式スプライト（UCL 2.0）のオレンジツインテール・水色衣装を「素の自己」として定義。オーブ未装備時は Aurora Teal アウトライン表示。

**職業切替方針: パーツ 3 点差替え（Sprite Library Variant）**

- Unity 2D Animation の Sprite Library Variant で腕・胴・頭アクセサリスロットのみ差替、換装コスト最小
- プレイヤーが「Unity-chan が剣士になっている」と読める（別人物ではなく変身）
- 参考: Mega Man X のアーマーパーツ集め × MapleStory の職業衣装差替

### B. Distinguishing Feature Rules

| 識別ペア | 第1識別子 | 第2識別子 | 第3識別子 |

| 敵 vs NPC | **スタンス**: 敵=戦闘前傾+武器、NPC=直立+武器なし | **目色**: 敵=赤橙、NPC=黒青 | NPC のみ外側白ハイライト 0.5px |
| 敵 vs プレイヤー | **向き**: 敵=画面右向き正面、Unity-chan=左向き標準 | **縁取り輝き**: Unity-chan のみ職業オーブ Glow | サイズと位置（Unity-chan 常に画面下 30-40%） |
| 敵の脅威度 | **サイズ階層** | **Glow 強度**（Tier2 アクセント、Tier3 Aura） | 色彩密度（Tier1 = 2-3色、Tier3 = 5-6色） |
| 職業ごと | **頭部形状** | **武器シルエット**（Section 3 Design Test） | **オーブカラー Glow** |

**脅威度エスカレーション**: 脅威が高いほど **形の鋭角密度と Glow 強度が増す**。Tier1 = 鋭角 1-2 点（Glow なし）、Tier2 = 鋭角 3 点以上（アクセント Glow）、Tier3 = 全方向突起 + 環境に滲む Aura。

### C. Expression / Pose Style

**方向性**: 「躍動するアニメ絵本」。固い ↔ **表情豊か寄り**。リアル ↔ **誇張寄り**（カートゥーン的ではなく、原神/テイルズ系の標準誇張）。Ori 無表情とは対極、MapleStory + 白猫の「キャラが生きている感」。

**戦闘時ポーズ指針**（Pillar 3 実装の核）:

| モーション | 重心・体の変化 | 表情 | 参考 |
|-----------|------------|-----|-----|
| 通常攻撃 | 前傾 + 重心低下 + 武器前方鋭角 | 戦闘表情（細目集中） | Mega Man X の腰が入った攻撃 |
| ジャンプ攻撃 | 空中で体が伸び → 着地でしゃがみ込み | 戦闘表情 → 一瞬「ふわ」安堵 | Ori の浮遊感 + 重め着地 |
| ダメージ受け | のけぞり + 後方傾き 20°以上 | ダメージ表情 | MapleStory のけぞり、白猫被ダメ |
| 職業切替 | 一瞬の直立静止 (0.05s) → 新職業スタンス変形 | 切替中は目閉じ or 輝く → 開いて新表情 | 原神キャラ切替の質感 |

**待機アニメーション（Idle）**: 3〜5 秒ループの小さな動き。呼吸（胸上下 2 サイクル/5s）、ツインテール微風揺れ（物理スプリング）、職業ごとの武器クセ動作（剣士=握り直し、弓士=弦を弾く、魔法使い=杖先が小さく光る）。

**長時間放置 Idle（30 秒以上）**: キョロキョロ、オーブを手で弄ぶ、しゃがんで見つめる等。State 8 ベンチ idle と同系統の生活感。

### D. LOD Philosophy

**2D 横スク メトロイドヴァニアの LOD は「距離ではなくレイヤー役割で決める」**

| レイヤー | 詳細度 | 縁取り | 密度 | 代表要素 |
|---------|--------|------|------|---------|
| 前景プレイヤー層 | 最高 | 2px + Glow | 高 | Unity-chan、敵 Tier 2-3 |
| 中景インタラクタブル | 中 | 1-2px | 中 | 敵 Tier 1、NPC、宝箱 |
| 遠景背景 Layer 2-3 | 低 | 0-0.5px | 低 | 建物シルエット、木、山並み |
| 最遠景 Layer 4-5 | 最低 | なし | なし | 空、霧、遠景崖 |

**Pixel Perfect Camera 統一ルール**:
- **基準解像度: 384×216px**（代替 360×224px）
- 前景キャラ: **48×48px 基準**（ボス 128×96〜256×192px）
- **Pixel Perfect Camera で Integer Scale のみ**（非整数スケール = 縁取りジャギー崩壊）
- AI 生成スプライトは必ず 48px グリッド手動整合
- 参考: Shovel Knight の 480×270 厳格 Pixel Perfect、ただしドット密度低めで「アニメ調ツルっと感」優先

**ボス戦詳細度**: 顔スプライトと体メインシルエットを分離、フェーズ変化時は顔のみ差替で演出（全身差替より工数削減）。

### E. AI 生成パイプライン協働（案 C Hybrid）

**AI 生成で効果が高い範囲**:

| カテゴリ | 用途 | 品質管理 |
|---------|-----|-------|
| 敵キャラ Tier 1-2 | 生物系・機械系バリエ量産 | Reference Sheet 固定、手動ピクセル整合 |
| 武器・衣装パーツ | 職業衣装の装飾バリエ | 48px グリッド + Deep Slate 縁取り後処理統一 |
| 背景小物・装飾タイル | 環境密度増加 | Zone カラールール（Section 4 D）厳守 |
| 表情差替スプライト | 感情表現拡張 | 顔切り取り、骨格アニメ非使用 |

**AI 生成で再現困難な範囲（人手必須）**:
- Unity-chan 本体骨格アニメーション（UCL 2.0 公式 rig 使用）
- ボス Tier 3 の連続モーション（多フレーム一貫性が AI の弱点）
- 職業切替エフェクト（Section 1 Principle 3）→ VFX Graph / Particle 手実装
- Color Wash / Glow の動的表現 → Shader 必要、technical-artist に委譲

| キャラ種別 | 必要参照画像 | 含む内容 |
|----------|-----------|-------|
| 敵 Tier 1 | 最低 3 枚 | 正面・横向き・攻撃モーション |
| 敵 Tier 2 | 最低 5 枚 | 上記 + ダメージ表情 + 特殊行動 |
| 敵 Tier 3 | 最低 8 枚 | 全フェーズのキーポーズ |
| NPC | 最低 4 枚 | 正面 + 表情 3 パターン |

**Reference Sheet フォーマット**: 背景 Sky Ivory、パレット見本右端、スケール定規（Unity-chan 48px）右下、AI プロンプト要点右上。

## 6. Environment Design Language

### A. Architectural Style と世界観

> *世界観・文明設定の詳細は narrative-director 領域。本 Section では「建築様式の根拠」として最小限言及し、narrative 確定後に詳細拡張する*

#### Zone 1: 王国遺跡（入門）— 「栄えたが去った文明」

**建築様式**:
- **積石工法**: 整然とした矩形切石ブロック主体（Section 3 の「整った四角」）
- **アーチ多用**: ロマネスク様式の半円アーチ（開口部・橋梁・窓）、接続部アーチと形語彙統一
- **彫刻装飾**: 柱頭・壁面に職業オーブのレリーフ（六角形モチーフ）
- **スケール**: 天井 4〜6 倍キャラ高、かつての偉大さを暗示

**参考**: Celeste の忘れ去られた神殿（整った石の静けさ）× Hollow Knight 旧都の残滓 × MapleStory Perion の「古代感 without 野暮ったさ」

#### Zone 2: 自然洞窟（中盤）— 「人工+自然の共存」

**建築様式**:
- **自然洞窟ベース + 人工介入痕跡**: 岩盤に石積み補強壁が一部残る、境界が曖昧
- **鍾乳石 + 朽ちた木製足場**: 不等辺三角形の鍾乳石（Section 3）間に朽ちた木製プラットフォーム
- **地下川・水面**: 水平線の自然プラットフォーム + 波紋 VFX
- **発光生態系**: 松明が消え、発光キノコ群落が照明を担う

**参考**: Hollow Knight 菌類の廃墟（ただし明度担保で「暗黒ダンジョン回避」遵守）× Ori 自然洞窟の発光素材温かさ × MapleStory Ellinia 地下

#### Zone 3: 古代神殿（終盤）— 「完全な幾何学」

**建築様式**:
- **誤差ゼロの幾何学完璧性**: 直角・円・多角形（Section 3 最極点）
- **浮遊プラットフォーム**: 床非接続の浮遊六角形（重力無視配置）
- **魔法陣床**: 全床に走る Void Indigo 発光文様
- **六角柱**: 丸柱ではなく六角柱（職業オーブの六角形 = 建築単位）

**参考**: 原神の古代遺跡エリアを 2D 横スクに圧縮 × Hollow Knight の「完璧すぎる古代部屋」感

#### ゾーン間「歴史感」の視覚ヒント

| 視覚ヒント | Zone 1 | Zone 2 | Zone 3 | 伝えること |
|-----------|--------|--------|--------|----------|
| 建材の精度 | 整った切石（人力） | 不規則岩 + 粗補強 | 完璧な幾何学（超技術） | 新旧の順序 |
| 職業シンボル | 壁レリーフ（3 種類） | 崩れた碑文 | 壁画（**7 種類**：DLC 伏線） | 元々もっと職業があった |
| 光源 | 太陽光・石窓 | 発光キノコ | 魔法陣発光（自律） | 文明の種類が違う |
| 損傷度 | 崩れ・ツタ | 木材腐朽・水流侵食 | 損傷なし（古代保存技術） | 最古が最も完全 |

### B. Texture Philosophy

**選択: Stylized Hand-Painted（手描き風フラット）**

| 軸 | 方針 |
|---|---|
| テクスチャ種別 | 手描き風フラットシェーディング、筆タッチ感 |
| PBR | 環境タイルには**不使用**。Glow/Bloom のみ URP Post-Processing |
| 輪郭線 | 全スプライト・プロップに **1-2px Deep Slate `#2C3E50`** アウトライン。背景遠景は輪郭なし |
| タイル解像度 | 前景 32×32 / 中景 48×48 / 背景 64×64px |
| タイル陰影方向 | 全タイル共通 **左上→右下** の固定ライティングをテクスチャベイク（視差で照明ぶれない）|
| テクスチャ密度 | 前景=高密度、中景=中密度、最遠景=低密度（単色+グラデーション）|
| ハイライト | 石材稜線に 1px 白（Sky Ivory）のリム、水面・発光素材のみウェット感追加 |

**AI 生成との相性**: Midjourney / Stable Diffusion の `2D game tileset, stylized, painterly` 系プロンプトと相性良好。Consistency Reference として「Parchment + Moss Stone の Zone 1 基準タイル」を 1 枚人力作成し img2img reference で固定運用。

**後処理チェックリスト**: (1) アウトライン 1-2px 手動追加、(2) ハイライト稜線確認、(3) 色を Hex パレットに補正

**参考**: MapleStory の「塗り絵のような背景」 + Ori 「背景素材の描き込み密度」。Celeste の Pixel Art 密度まで落とさない（ステッカー比喩は滑らかな輪郭を要求）。

#### フレーバーテキストとの連動（Pillar 2）

プロップ発見 → インタラクト時にフレーバーテキスト 1〜2 行が補足。例:

#### プラットフォームリズム（コンボ移動誘導）

```
通常移動（等間隔）:
[■■■]    [■■■]    [■■■]
         約 1.5 キャラ幅ギャップ

コンボ誘導（密→疎加速型）:
[■■■] [■■■][■■■]          [■■■]
  密              疎（空中コンボ強要）

**参考**: Celeste がプラットフォームリズムで「ダッシュを使え」を教える手法と同じ、「ここで職業切替を使うと気持ちいい」を形で教える。

## 7. UI/HUD Visual Direction

> *art-director と ux-designer の並列レビューで 4 コンフリクトを抽出、全両立解採用 + Color Wash から HUD Canvas 除外を確定*

### A. Diegetic vs Screen-Space — **Semi-Diegetic 採用**

スクリーンスペース固定 + 世界語彙の衣（六角形・丸角）。

- **理由**: ポップ・ダンジョン世界では完全 Diegetic は視認性悪化。Pillar 1 の「跳ねる切替インジケーター」がカメラ揺れと干渉しないよう Screen-Space 固定

### D. UI Animation Feel

#### 画面遷移

| 種別 | 採用方針 | タイミング |
|------|--------|---------|
| 画面間遷移 | 六角形タイル展開（中央から広がる） | 0.25-0.35 秒 |
| ポーズメニュー出現 | 上から「ゆっくり落ちる」(ease-out) | 0.2 秒 |
| ポーズメニュー退場 | 上に「すっと引き上がる」(ease-in) | 0.15 秒 |
| 画面内ポップアップ | 中央から「ぽんっ」拡大 (overshoot 0→1.1→1.0) | 0.2 秒 |
| ダイアログ退場 | 下に「すっと下がる」(ease-in) | 0.15 秒 |

Hollow Knight の「静かな淡いクロスフェード」とは対照的に、本作は微細な弾力感を持たせる。0.3 秒以上は「もっさり感」を生むため上限。

#### ボタンフォーカス（Gamepad d-pad 対応）

| 状態 | 視覚変化 | タイミング |
|------|--------|---------|
| 非選択 | UI Neutral、scale 1.0 | — |
| **フォーカス** | UI Emphasis 縁取り出現 + scale 1.05 (ease-out) | 0.1 秒 |
| 押下瞬間 | scale 0.95 (squish)、輝度 +20% | 0.05 秒 |
| 押下解放 | scale 1.0 + 短い Glow フラッシュ | 0.1 秒 |
| 非活性 | 彩度 0% + opacity 40% | — |

#### オーブ HUD アニメーション（Pillar 1 の核実装）

**通常待機（Idle）**:
- 現在職業オーブ: 4 秒周期で scale 1.0 → 1.08 → 1.0 の「息継ぎ」パルス（ease in-out）
- 待機職業オーブ: 無動作（メイン職業の鼓動を強調）

**切替トリガー直前（予備動作）**:
- 次職業オーブ: 0.05 秒で scale 1.0 → 1.15 の「準備膨らみ」
- 切替ボタン長押し検知時: Glow 輝度 50% → 100%

**切替確定瞬間（フレーム別、60fps 基準）**:

```
f0:  旧職業オーブ scale 1.0 → 0.85 (squish down)
f3:  新職業オーブ scale 0.0 → 1.25 (pop in + overshoot)
f5:  新職業オーブ scale 1.25 → 1.0 (settle, ease-out)
f6:  旧職業オーブが待機スロットへ「飛ぶ」(x, y アニメ 0.12 秒)
f12: 全スロット scale 1.0 で安定

同時:
  新職業オーブの Glow 0 → 最大輝度 → 通常輝度 (0.3 秒フェード)
  Section 2 State 3 の Color Wash とタイミング同期
```

**Orb Acquisition（新職業取得）**:
- 新スロットが y -40px から ease-out で着地
- 着地時 scale 1.0 → 1.3 → 1.0 の「ドスン」バウンス
- 着地後 Glow が 3 秒かけてフェードインして定常輝度に収まる

#### ⚠️ Conflict 2 両立解: UI モード中は HUD オーブ静止

**ゲームプレイ中のみ HUD オーブの跳ねアニメーションを有効化**。ポーズ・メニュー・ダイアログ表示中は UIモードフラグで制御し、HUD オーブを静止または非表示にする。「切替が花になる」演出はゲームプレイのフィードバックであり、メニュー操作の UX ノイズにしない。

#### ⚠️ 重要仕様: Color Wash (State 3) から HUD Canvas を除外

**HUD Canvas は Color Wash の Render Texture マスクから外す**。被ダメ + 切替が重なる高頻度シナリオで「HP が 0.1-0.4 秒見えない」事態を防ぐ。「切替が唯一の主役」というアート意図は世界空間（キャラ、背景）でのみ実現し、HUD の機能は守る。

#### ダメージポップアニメーション

```
生成 → y +20px の上昇 (ease-out, 0.5 秒)
    → opacity 100% → 0% フェードアウト (最後の 0.2 秒) → 消滅

通常: scale 1.0 + 通常上昇
クリティカル / コンボ上限: scale 1.5 + 水平広がり (x scale 1.2) + 輝度 +30%
同時多段ヒット: 前ポップを若干右にオフセットしてスタック（最大 3 つ）
```

       scale 0 → 2.0 (overshoot 2.3 → 2.0)
       Glow: 職業カラー 100% → 70% フェード
       六角形内: 職業シンボルが fade-in

[0.8s] 画面中央に職業名テキスト（H1 Display / Dela Gothic One）
       例: 「剣士の魂、宿る」「弓士の息吹、目覚める」
       テキストが上から「ふわっと降りてくる」(ease-out, 0.4 秒)

[1.5s] 説明テキスト 2 行（Body フォント、フェードイン 0.3 秒）
       例: 「R1/L1 で職業を切替える。剣士の力で岩を砕く。」

[2.5s] 六角形が縮小して HUD の新スロット位置へ「飛んでいく」
       — ベジェ曲線軌跡 — duration 0.5 秒
       HUD 再表示、新オーブが「ドスン」バウンス着地

[3.0s] 通常ゲームプレイ復帰、0.5-1 秒の余白（自動進行しない）
```

**スキップ**: A / Enter で可能、ただしデフォルトはフル再生（2 周目以降向け）。

#### E-5. NPC 会話 vs フレーバーテキスト分離

**NPC 会話ウィンドウ**:
- 画面下 20% を占める横長パネル、丸角矩形 (radius 8px)、Sky Ivory 85% opacity
- 左: NPC バストアップ (64×64px ベース、2 倍表示)、感情 3 段階差替
- 右: テキスト（Noto Sans JP Regular 18px、タイプライタ式 1 文字 40-50ms、長押しで即表示）
- 下右: 「[A] 次へ」「[B] 閉じる」コントロールヒント

**フレーバーテキスト（オーブ・碑文インタラクト時）**:
- 画面中央上寄りの「碑文スタイル」ウィンドウ
- 形状: **六角形フレーム**（ゲート形語彙統一）
- 背景: Neutral Mid 40% opacity（「古さ」暗示）
- フォント: 同 Noto Sans JP だが italic 風の傾き
- 表示速度: 「刻まれる」速度（1 文字 80ms、遅め）
- 閉じ方: フォールアウト（opacity 100% → 0%、0.5 秒後）

→ **「NPC が話す（現在）」と「碑文が語る（過去）」の時制を形で区別**

#### ⚠️ Conflict 3 両立解: 戦闘中はフレーバーテキスト発動を無効化 / キュー保留

敵が存在する部屋でのプロップインタラクトは **「戦闘中は無効化」または「戦闘終了後に再生」**。プレイヤーが立ち止まれる状態でのみテキストが発動する。Section 6「テキスト先行で読ませるゲームにしない」設計と整合。

#### E-6. 確認ダイアログ

- 画面中央モーダル、丸角矩形 (radius 8px)、Sky Ivory 90% opacity
- サイズ: 320×160px（1080p 基準）
- 構成: H3 テキスト + ボタン 2 個（YES/A、NO/B）
- **破壊的アクション**: UI Danger `#FF4D4D` 縁取り
- **安全側**: デフォルトフォーカス（UI Emphasis 縁）
- 背景: ポーズメニューのブラー処理を継承

### F. Gamepad-First UI Navigation

- 全 UI 画面 **Explicit Navigation**（自動 Navigation NG、複雑レイアウトで失敗しがち）
- **フォーカスリング二層構造**:
  - 第一層（静的）: 選択中要素の縁取り外側に UI Emphasis `#FFED8A` で 2px アウトライン、形状継承（六角形ボタン → 六角形リング）
  - 第二層（動的）: 0.8 秒周期のゆっくりしたパルス。**戦闘中 HUD パルスなし**（HUD は基本操作対象にならない）
- **フォーカス時 scale 1.0→1.05**（Fitts の法則、視認性 + ターゲット明確化）
- **SE**: 1 種のニュートラルなチック音（決定音と区別、オーブ共鳴音風）
- **d-pad オートリピート**: 初回 150ms 遅延 + 80ms 間隔
- **スティック**: 0.3 デッドゾーン
- **フォーカスグループ視覚境界線**: Sky Ivory 半透明パネル + わずかな明度差
- **1 画面 8 要素上限**（プログレッシブ・ディスクロージャー）
- **フォーカスの記憶**: サブメニューから戻った際、親画面の最後フォーカスを復元

### G. HP Low 多チャンネル警告（Conflict 4 両立解）

**色だけに依存しない 4 チャンネル同時発動**:

1. **動き**: HP バー枠の点滅（1Hz ゆっくり）
2. **空間**: 画面四隅のヴィネット（赤いパルス、UI 要素と空間競合せず）
3. **音**: 警告 SE
4. **触覚**: コントローラー振動（Gamepad 限定）

**HP バー塗りの赤グラデーションと点滅赤の視覚分離**: 「**静的赤＝剣士色**」vs「**動的赤点滅＝危険**」をモーションで区別。HP Low 状態の点滅は HP バー自体ではなく **HP バー周囲枠（縁）** で表現することで、残量表示（静的）と危険警告（動的）を完全分離。

- **HUD 配置**:
  - 左上三角: HP バー（画面幅 20-25%、高さ 2.5-3%）

## 8. Asset Standards

> *art-director（規約方向性）+ technical-artist（Unity 6.3 ハード制約）の並列レビューで 5 コンフリクト抽出、全推奨解採用済み*

### A. ファイル形式

| カテゴリ | 形式 | 設定 |
|---------|-----|-----|
| スプライト（キャラ・敵・小物） | PNG-32 | RGBA 32bit、αプレマルチプライ NG |
| スプライトシート（4×4 等） | PNG-32 | metadata は Unity Sprite Editor の Multiple Mode で管理 |
| 背景タイル | PNG-32 | 不透明部分は DXT1 圧縮、透過は DXT5 |
| UI アイコン | PNG-32 | raster 基準（SVG は Tier 3 以降検討） |
| アニメーションデータ | Unity Animator Controller + Sprite Library Asset | C# event でゲームコードと連携 |
| マテリアル | `.mat`、URP Sprite-Lit-Default 基準 | カスタムは Shader Graph 経由 5 種上限（後述） |
| フォント | TTF / OTF → TextMeshPro Font Asset 化 | Static Atlas 事前生成（ランタイム生成 NG）|

**Pixel Art の圧縮ポリシー（重要）**: 16px 以下のアイコン・1-2px 縁取りスプライトは **Compression None / RGBA32** を例外許可（DXT/BC 圧縮で縁取り 1-2px が崩れる）。Celeste / Dead Cells も同方針。VRAM コストは Texture Memory Budget 内で管理。

### C. テクスチャ解像度ティア

#### スプライト原寸基準（Pixel Per Unit = 100、PSD と PNG で統一）

| カテゴリ | 原寸 px | 実機表示 | Atlas 配置 |
|---------|--------|---------|----------|
| 前景キャラ | 48×48 | 48-96px (Pixel Perfect Scale) | `Atlas_Char_[Class].spriteatlas` |
| ボス | 128×96 〜 256×192 | そのまま | `Atlas_Boss_[Name].spriteatlas` |
| 敵 Tier 1 | 32×32 〜 48×48 | そのまま | `Atlas_Zone[N]_Enemies.spriteatlas` |
| 敵 Tier 2 | 64×64 〜 96×64 | そのまま | 同上 |
| 敵 Tier 3 | 128×96 〜 256×192 | そのまま | `Atlas_Boss_*` または個別 |
| タイル前景 | 32×32 | そのまま | `Atlas_Zone[N]_Tile.spriteatlas` |
| タイル中景 | 48×48 | そのまま | 同上 |
| タイル背景 | 64×64 | そのまま | 同上 |
| UI アイコン | 32×32 キャンバス | 24-48px | `Atlas_UI_HUD/Menu.spriteatlas` |
| HUD オーブ | 48px / 32px | そのまま | `Atlas_UI_HUD.spriteatlas` |

**Power-of-Two ルール**: 個別テクスチャは小さく保ち（最大 256-512px）、**Atlas 側で 2048×2048 まで集約**（SRP Batcher のバッチ統合のため）。

**Mip Map**: 2D Pixel Perfect では基本 OFF（VRAM 無駄消費 + ピクセル補間ぼけ）。視差スクロール最遠景 Layer のみ ON 検討（Tier 2 以降）。

### D. LOD レベル（複数解像度バリアント）

| レイヤー | 解像度 | 縁取り | LOD 戦略 |
|---------|------|------|--------|
| 前景プレイヤー層 | 48×48 (基準) | 2-3px Shader Outline + Glow | 高密度、職業別 Atlas |
| 中景インタラクタブル | 32×32 〜 64×64 | 1-2px Shader Outline | 中密度 |
| 遠景背景 Layer 2-3 | 64×64 | 0-0.5px (焼き込み or なし) | 低密度 |
| 最遠景 Layer 4-5 | 単色 or 2 段グラデ | なし | 最低密度 |

**整数倍スケールルール**: 全スプライトは Pixel Perfect Camera で **Integer Scale のみ**（非整数スケール = 縁取りジャギー崩壊、絶対 NG）。

### E. エクスポート設定（Importer 規約）

#### PSD Importer 設定（Unity-chan 公式素材想定）

| 設定 | 値 | 理由 |
|-----|---|-----|
| Import Mode | Individual Sprites (Character Rig) | 2D Skeletal Animation のボーン紐づけ |
| Pixels Per Unit (PPU) | **100** | Pixel Perfect Camera 基準 |
| Mosaic（レイヤー Atlas 生成） | **ON** | PSD の各レイヤーを Atlas 自動梱包 |
| Filter Mode | **Point (no filter)** | Pixel Art 補間ぼけ防止 |
| Generate Mip Maps | **OFF** | 2D Pixel Perfect では不要 |
| Compression | **DXT5 (BC3)** | 半透明レイヤー含むため |
| Layer 分割 | ボーン部位単位（Head, Torso, ArmL/R, LegL/R, WeaponSlot） | 衣装差替対応 |
| Sprite Mode | Multiple（自動） | — |

#### 通常 PNG Importer 設定

| 設定 | 値 | 理由 |
|-----|---|-----|
| Texture Type | Sprite (2D and UI) | — |
| Sprite Mode | Multiple (シート) / Single (単体) | — |
| Pivot | **Bottom Center** | 物理コライダー原点を底辺に |
| Filter Mode | **Point** | 必須 |
| Generate Mip Maps | **OFF** | 不要 |
| PPU | **100** | PSD と統一 |
| Alpha is Transparency | **ON** | 半透明全般 |
| Compression | 不透明=DXT1 / 半透明=DXT5 / 16px以下=RGBA32 | 縁取り保護 |
| Max Texture Size | キャラ 256 / タイル 512 / ボス 512 | Atlas で集約 |
| Wrap Mode | **Clamp** | 端からの色滲み防止 |

#### Sprite Atlas v2 Packing 設定

- Padding: **2px**（Point Filter 時の境界滲み防止）
- Allow Rotation: **OFF**（Pixel Art の崩れ防止）
- Tight Packing: **OFF**（Allow Rotation OFF 前提では恩恵なし）
- Include in Build: **OFF**（Addressables 管理下のため）

#### Sprite Mesh Type の使い分け

| カテゴリ | Mesh Type | 上限 |
|---------|-----------|-----|
| タイル | **Full Rect** 強制 | SRP Batcher 統合のため |
| UI アイコン | **Full Rect** 強制 | Canvas Rebatch 抑制 |
| キャラクター（Skeletal） | **Tight** + Tessellation Tolerance 0.5+ | 256 頂点（主人公）/ 128（ボス）/ 64（敵） |
| ボス（128-256px） | **Tight**、最大 64 三角形/スプライト | 多パーツ Skeletal 構成可 |
| 敵 Tier 1-3 | **Tight**、最大 32 三角形/スプライト | 同時複数体想定 |
| VFX スプライトシート | **Full Rect** | Tight メッシュ再生成コスト回避 |
| 背景小物 AI 生成 | **Full Rect** 基本、透明領域>50% のみ Tight 16 三角形上限 | パイプライン安定性優先 |

### G. ハード制約とパフォーマンスバジェット（technical-artist）

#### G-1. ポリ数 / 三角形バジェット

- **全画面合計スプライトメッシュ三角形数上限: 8,000**
  - キャラ系 2,000 / タイル系 3,000 / 敵 1,500 / VFX 1,000 / UI 500
- **警戒ライン: 6,000 三角形**（Profiler で黄色アラート）
- 2D Skeletal: Bone Weight 4 ボーン/頂点（デフォルト）、Tessellation Tolerance 0.5+

#### G-2. テクスチャメモリバジェット（VRAM 1.5GB 内訳）

| カテゴリ | 割り当て |
|---------|--------|
| ゲームプレイテクスチャ | **600 MB** |
| エンジンバッファ（Render Texture, Depth, GBuffer） | 200 MB |
| UI テクスチャ | **100 MB** |
| VFX テクスチャ | **150 MB** |
| 予備・OS/ドライバ | 450 MB |
| **合計** | **1,500 MB** |

**ゾーン 1 セットあたり目安**: 15-20 MB（**上限 32 MB**、二重保持ピーク考慮）。

#### G-3. Sprite Atlas v2 規約

- **1 Atlas 最大 2048×2048px**（Steam Deck 統合 GPU 制約、4096 非推奨）
- 同一シーン同時表示スプライトは必ず同一 Atlas に
- Atlas 分割（H-5 推奨解採用）:
  - `Atlas_Zone[N]_Tile` / `Atlas_Zone[N]_Prop`（**ゾーンごとに 2-3 枚許容**、Draw Call 増加を許容範囲で受け入れ）
  - `Atlas_Char_[ClassName]`
  - `Atlas_UI_HUD` / `Atlas_UI_Menu`
- Include in Build: **OFF**（Addressables 管理）

#### G-4. 圧縮形式選択

| 条件 | 形式 |
|------|-----|
| 不透明タイル（背景・地形） | DXT1 (BC1) |
| 半透明スプライト（キャラ・敵・UI） | DXT5 (BC3) |
| 精細グラデ VFX | BC7（VFX 150MB 内） |
| Pixel Art 16px 以下アイコン・縁取り保護必要 | **RGBA32 非圧縮** 例外許可 |
| Crunched DXT | **禁止**（解凍 CPU コスト） |

#### G-5. Draw Call バジェット（合計 300 以下、SRP Batcher）

| カテゴリ | 上限 DC |
|---------|--------|
| タイル/背景（バッチ済み） | 30 |
| キャラ（主人公+職業） | 10 |
| 敵（同時最大 8 体） | 40 |
| **VFX パーティクル** | **60**（職業切替最優先 10 DC 確保） |
| **2D Lights** | **80** |
| UI（HUD + メニュー） | 30 |
| その他 | 50 |

#### G-6. VFX & 2D Light（H-2/H-3 推奨解採用）

**VFX**:
- 同時アクティブ ParticleSystem 上限: **20 個**
- **職業切替エフェクト最優先**（10 DC 確保）、環境アンビエントは残枠
- Material/Texture 共有で Atlas 経由バッチ統合（URP/Particles/Standard Unlit ベース）

**2D Light**:
- 1 シーン内 2D Light 上限: **12 個**（Global 1-2、Point/Spot 6、Free Form 4）
- Shadow Casting Light: **最大 4 個**
- **Emissive テクスチャ + 1 Global Bloom Pass で代替光源を多用**（実 Light は劇場ボス・主光源のみ）

#### G-7. Material / Shader 多様性

- **デフォルト**: URP Sprite-Lit-Default
- **職業色 4 種は MaterialPropertyBlock で単一 Material 共有**（職業切替時 `_Color` を SetColor、SRP Batcher 維持）
- **カスタムシェーダー上限 5 種**:
  - `Sprite_Outline.shadergraph`（Shader Outline、主役系のみ適用）
  - `Sprite_Emissive.shadergraph`（オーブ Glow）
  - `Sprite_FlashHit.shadergraph`（ダメージフラッシュ）
  - `Sprite_Dissolve.shadergraph`（敵消滅）
  - `Sprite_ColorWash.shadergraph`（ゾーン別カラーウォッシュ）

#### G-8. 縁取りシェーダー実装方針（H-1 推奨解採用）

**B + 焼き込み混合戦略**:

| カテゴリ | 縁取り実装 |
|---------|---------|
| 主役（プレイヤー、敵、ボス、インタラクトオブジェクト） | **Sprite_Outline.shadergraph で Shader Outline**（+1 Pass/スプライト） |
| タイル、背景小物、最遠景 | **スプライト素材へ手動焼き込み**（Draw Call 増加なし） |
| UI アイコン | 既に手動描画前提 |

→ Draw Call 予算 300 内に収める。修正コストは Shader 適用範囲が主役のみのため低い。

#### G-9. Addressables アセットメモリ管理（Addressables 2.0）

**Scene 単位ロード**:
- 1 部屋 = 1 Addressable Scene
- 部屋遷移: 廊下中に次部屋を `LoadSceneAsync` 非同期プリロード → 遷移完了後旧 Scene を `UnloadSceneAsync` → `Resources.UnloadUnusedAssets()` + `GC.Collect()` を非同期分散
- **`SceneLoadingException` を try/catch でフォールバック UI 必須**（Unity 6.3 で例外スロー化）

**Group 分割**:

| Group | 内容 | ロード戦略 |
|-------|-----|---------|
| `AlwaysLoaded` | 主人公スプライト、共通 UI Atlas、コア SE | 起動時常駐 |
| `Zone_[N]_Tiles` | タイル Atlas、背景 Atlas | ゾーン入場時、退場後アンロード |
| `Zone_[N]_Enemies` | ゾーン N の敵 Atlas | 同上 |
| `Zone_[N]_Props` | 背景小物・AI 生成差分 | 同上 |
| `Class_[ClassName]` | 職業 Variant（衣装差分） | 職業解放時、以降常駐 |
| `Boss_[BossName]` | ボスグラフィック | ボスエリア入場直前 |
| `UI_Menus` | メニュー、ポーズ画面 | 初回メニュー開時、以降常駐 |

**メモリ常駐上限**:
- 常駐（AlwaysLoaded + 解放済み Class_*）: 最大 **128 MB**
- アクティブゾーン: 最大 **64 MB**
- プリロード次ゾーン: 最大 **32 MB**
- **合計ピーク: 224 MB**

#### G-10. AI 生成アセットの Variant 管理

- Sprite Library Asset の個別カテゴリで管理
- 命名: `SpriteLib_Class_[ClassName]_[BodyPart]` → ラベル `[Variant_N]`
- 職業切替時: `SpriteLibraryContainer.spriteLibraryAsset` をスクリプトで差し替え（Animator.Play() の 1 フレーム遅延を迂回、即時切替実現）
- AI 生成 Variant は `Zone_[N]_Props` Group に梱包

#### G-11. パフォーマンス検証チェックリスト

| 計測対象 | 目標 | 警戒 | ツール |
|---------|------|-----|------|
| 全体フレーム時間 | 16.6ms | 18ms | Unity Profiler |
| Update ループ | ≤4ms | 5ms | CPU Usage |
| Physics2D (Box2D v3) | ≤2ms | 3ms | Physics Profiler |
| GPU Rendering | ≤6ms | 8ms | GPU Usage / Render Graph Viewer |
| Draw Call | ≤300 | 270 | Frame Debugger |
| Texture VRAM | ≤600MB | 500MB | Memory Profiler |
| 全体 RAM | ≤1024MB | 800MB | Memory Profiler |
| GC Alloc/フレーム | 0B 定常 | 1KB | CPU Profiler |

**Steam Deck（Zen2 + RDNA2、TDP 10-15W）想定**:
- CPU 4C/8T 3.5GHz、Box2D v3 マルチスレッド恩恵あり
- GPU RDNA2 8CU、384×216 → Integer Scale ×5 = 1920×1080 ネイティブ
- 想定フレームタイム: PC 16.6ms 設計で Steam Deck 20-22ms 前後（推測、実機検証必須）
- 60fps ロック厳しければ 40fps（VRR）fallback 検討

## 9. Reference Direction

> *4 リファレンスを別軸でキュレート。各リファレンスは「ジャンル慣習 → 環境照明 → キャラ造形 → 誘導と演出」の 4 軸を独立に担当する。*

### Reference 1: MapleStory（2D 横スクロール ARPG、Nexon 2003〜）

**担当軸**: ジャンル慣習（職業色 / 縁取り / エフェクト彩度）

#### Take（取り入れる）
- **キャラクター縁取りの読みやすさ設計** — 背景の明暗に関わらずキャラが視認できる黒/濃色アウトライン。本作の Deep Slate `#2C3E50` 1px @ 1:1 縁取りの直接的な参照元
- **職業ごとの色サイン** — 剣士=赤、弓士=緑/茶、魔法使い=青/紫という社会的共通理解。本作の Ember Crimson / Forest Emerald / Void Indigo がここから引き継ぐ
- **高彩度+白飛び強調のポップ感** — 攻撃エフェクトで彩度最大まで振り切る表現を「ご褒美」として限定使用。Combo-Brightness Escalation の参照先

#### Avoid（避ける）
- UI の情報密度（MapleStory は極端に高密度で現代 UX とは乖離）
- キャラの顔比率（SD 体型の誇張、本作は Unity-chan ベースの等身）
- 過剰なパーティクル数（本作 Draw Call 300 制約と矛盾）

#### 接続するピラー / 原則
- Section 1 Principle 4: Combo-Brightness Escalation
- Section 4: 職業色定義
- Section 5 / Section 8: Outline 設計

### Reference 2: ジブリ背景画 / 『魔女の宅急便』美術監督・串田達也（Studio Ghibli、1989）

**担当軸**: 環境照明（生活感ある光 / 暖色陰影 / 背景の余白）

#### Take
- **「生活感ある光」の表現** — 窓から差し込む斜光、石畳の照り返し。光源を「演出」ではなく「世界の一部」として描く。Section 6 の Warm Light Direction（上 45°）と Zone 1 王国遺跡の雰囲気設計に直接適用
- **背景の「余白」ルール** — キャラが動く空間として意図的に中景を空ける構図設計。視差スクロールで Layer 1（中景）が混雑しないことの論拠
- **暖色ベース×寒色影のカラーペア** — Sky Ivory / Warm Shadow の組み合わせはこの手法の直系。影を「暗くする」のではなく「色を変える」

#### Avoid
- 水彩ブラシ質感のテクスチャ（本作はクリーンなフラット+手描きハイブリッド、水彩にじみは世界観から外れる）
- 実写に近い遠近感（本作は 2D 横スクロール視点で消点固定）

#### 接続するピラー / 原則
- Section 4: Color / Lighting Direction
- Section 6: Environment Zone 設計（特に Zone 1）
- Section 1: 世界の「呼吸」感（Sticker + Sunlit）

### Reference 3: loundraw（イラストレーター、「ミオリネ」「さよならの朝に約束の花をかざろう」メインビジュアル等）

**担当軸**: キャラ造形（小さいスプライトで感情を伝える / 表情可読性 / 発光管理）

#### Take
- **発光体を「白飛び手前」で止める階調管理** — 光るオーブ・魔法エフェクトの色設計で「白く飛ばしきらない、でも輝いている」を成立させる技法。Section 4 の Orb White を「純白にしない」決定と接続
- **キャラクターの表情可読性** — 小さいサイズ（ゲーム中の 48×48px 前後）でも目の輝き・口角で感情が読めるアイポイント設計。Section 5 の表情 4 モード（待機/戦闘/勝利/ダメージ）の造形指針
- **ソフトな陰影と平塗りの混在** — セル塗り + グラデ影を 1:1 で混在させる手法（全面グラデ塗りでも全面フラットでもない）

#### Avoid
- 成人向け・複雑な心理描写を前提にした繊細な表情演技（本作のキャラは 4 パターンの記号的表情で完結）
- ポスタービジュアル的な構図の複雑さ（ゲームスプライトは 48×48px 制約内で機能する必要がある）

#### 接続するピラー / 原則
- Section 5: Character Design / 表情システム
- Section 4: Orb White / エフェクト光量設計
- Section 1 Principle 1: Dual-Read Silhouette（表情で「待機 vs 戦闘」を判別）

### Reference 4: 『Ori and the Blind Forest』（Moon Studios、2015）

**担当軸**: 誘導と演出（光誘導 / 視差速度 / エモーショナルカラーシフト）

#### Take
- **光で誘導する「見えないマーカー」設計** — 進行方向にだけ白系の光を置く。HUD の矢印や地図なしで「次はあそこへ行く」を伝える。Section 6 F「光源配置で進む方向を明るく」の実装済み参照
- **視差スクロールの速度レシピ** — Layer 0（最遠）0.1x / Layer 1 0.25x / Layer 2 0.5x / Layer 3（前景）1.0x。本作の 4 Layer 設計の速度値の参考基準
- **エモーショナルカラーシフト** — 危機場面で世界全体が寒色化、回復場面で暖色化。本作の Class-Switch Color Wash の「一時的な世界の染色」表現に直接対応

#### Avoid
- HDR 光量（本作 sRGB 固定、Ori の過剰なブルーム処理は本作の Draw Call 予算と非互換）
- 流体的なパーティクル挙動（本作のエフェクトは URP 2D Renderer + 手書き Sprite Animation が基本、VFX Graph 大量パーティクルは不使用）

#### 接続するピラー / 原則
- Section 1 Principle 3: Class-Switch Color Wash
- Section 6 F: 光誘導、視差スクロール 4 Layer 速度設計
- Section 7: Light-as-Guidance の UI 補完

### キュレーション意図

各リファレンスが「別の方向」を指す独立軸を担当する構造:

| Reference | 担当軸 | 他と重複しない理由 |
|---|---|---|
| **MapleStory** | ジャンル慣習（職業色 / 縁取り / 彩度爆発） | 本作と同じ「職業切替 ARPG」の先行事例。ジャンル慣習を押さえるのはここだけ |
| **ジブリ背景** | 環境照明（生活感ある光 / 暖色陰影 / 余白） | 「世界の空気感・照明設計」軸。Map / Zone の質感に直結 |
| **loundraw** | キャラ造形（発光管理 / 表情可読性 / 陰影造形） | 「小さいスプライトで感情を伝える」軸。背景でも職業でもなくキャラ品質の基準 |
| **Ori and the Blind Forest** | 誘導と演出（光誘導 / 視差速度 / カラーシフト） | 本作最重要の Class-Switch Color Wash を「実装済みのゲームで確認できる参照」として位置づけ |

これら 4 件は「ジャンル慣習 → 環境照明 → キャラ造形 → 誘導と演出」という 4 つの独立した問いに、それぞれ別の先生が答える構成。どれか 1 件が欠けると、その軸についての判断基準が失われる。

### Reference の使い方（チーム運用）

- **新規アセット作成時**: 該当軸のリファレンスを開いて Take/Avoid を再確認してから着手
- **判断に迷った時**: 4 軸のうち最も近い軸のリファレンスに照合
- **AI 生成プロンプト作成時**: 該当 Reference のキーワードを参考に（例: loundraw 風の発光、Ori 風の光誘導等）。ただし生成物が「コピーしている」と読まれないよう Avoid 項目を必ず守る
- **新リファレンス追加時**: 既存 4 軸と重複しない「第 5 の軸」が必要。安易な追加は焦点をぼかす

### 保留事項

1. 北斎（浮世絵）の **造形ロジック軸** はカット判断したが、Tier 2 以降で「シルエット読みやすさ」のレビューが必要になった場合は再検討余地あり
2. 4 リファレンスはいずれも非ホラー・明るい / ポップ系で揃え、Anti-Pillar（ホラー回避）遵守
3. 各 Reference の具体的画像・スクリーンショット参照集は art-director で別資料化予定（本 Art Bible には文章のみ）
